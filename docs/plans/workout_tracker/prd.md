# Product Requirements Document - goal-tracker-v2

**Author:** Ana
**Date:** 2026-03-22

## Executive Summary

Goal Tracker will expand into a unified goal-and-workout web experience where completing a workout session creates a structured workout entry, including start and end timestamps, directly linked to goal progress. The product is designed for users who want consistent training outcomes without spending excessive effort on planning and logging overhead. The core problem addressed is that tracking itself often becomes friction: existing apps are feature-bloated, spreadsheets are tedious, and paper workflows are unreliable.

The intended outcome is a workflow where users focus on execution at the gym while the system handles session capture and progress linkage with minimal interaction cost. This integration is timely because the user has already demonstrated sustained behavior (45 unique tracked workout days out of 80 this year), indicating high potential value from reducing friction in the current tracking loop.

### What Makes This Special

The differentiator is direct coupling between workout sessions and goal tracking in one planning loop, rather than separate fitness and goal-management tools. The product records workout session boundaries (start and end datetime) and turns completed sessions into persistent workout entries that immediately contribute to goal tracking context.

The core insight is that adherence improves when logging burden decreases. By optimizing for faster, connected capture instead of broad feature breadth, the product positions itself as ready to use for consistent athletes: one flow from workout execution to goal progress, with minimal cognitive and operational overhead.

## Project Classification

- Project Type: Web App
- Domain: General (fitness/workout use case)
- Complexity: Medium
- Project Context: Brownfield

## Success Criteria

### User Success

Users can complete a full workout-tracking loop inside Goal Tracker without leaving the app context: start a workout session, log sets during the session, end the session, and view the resulting workout entry connected to goal-tracking context. Users can also manage training plans, browse exercise history, view progress graphs, and use a rest timer from a coherent mobile-first navigation flow.

The user success moment is reduced cognitive and operational overhead during training: users spend less effort on planning and logging and more effort on workout execution. Completion for MVP is defined as successful in-app session capture plus visible post-session history linkage using frontend mocked data.

### Business Success

In the first 3 months post-release of the integrated frontend experience, success is demonstrated by measurable adoption of workout workflows within Goal Tracker usage patterns, indicating that integration improves product stickiness and daily/weekly engagement depth.

In the first 12 months, success is demonstrated by sustained behavior where workout logging and goal tracking are used together as a single habit loop, validating the strategic value of unifying both domains in one product surface.

### Technical Success

The frontend implementation preserves existing Goal Tracker stability while introducing the workout module with API-ready architecture boundaries: mocked data providers are isolated and replaceable by real API clients without major UI rewrites. All new pages and flows are responsive, type-safe, and maintainable under current project conventions.

Technical quality for this PRD includes passing lint/type checks for changed frontend code, clear state and data-flow boundaries for future backend integration, and acceptable interaction performance for session logging and navigation on typical modern mobile and desktop browsers.

### Measurable Outcomes

- At least 80% of test users can complete start-to-end workout session logging flow without guidance in under 2 minutes.
- At least 70% of active users who access workout pages in a week complete at least one saved workout session in the same week.
- At least 60% of users who log workouts also interact with at least one goal-tracking surface in the same week.
- 100% of workout pages in MVP are mobile-responsive with no blocking visual regressions in target breakpoints.
- 0 blocking lint/type errors introduced by the workout frontend integration branch.

## Product Scope

### MVP - Minimum Viable Product

Integrate the workout-tracker frontend experience into Goal Tracker frontend with mocked data only, including: session-oriented workout logging flow, plans and plan detail interactions, exercise management screens, history and graph views, timer page, and shared navigation/layout integration aligned with Goal Tracker UX baseline.

MVP includes explicit API-ready seams (mock repositories/services/adapters) so backend replacement can happen in a subsequent PRD and implementation cycle without restructuring page-level UI logic.

### Growth Features (Post-MVP)

Backend-connected persistence, multi-device synchronization, richer analytics and progress insights, enhanced search/filter capabilities, session templates, smarter plan recommendations, and deeper cross-linking between goals and workout performance trends.

### Vision (Future)

A fully unified training and goal operating system where users plan goals, execute workouts, capture objective session data, and receive actionable progression feedback in one continuous loop with minimal manual overhead and high long-term adherence support.

## User Journeys

### Journey 1: Primary User (Athlete) - Success Path

Ana opens Goal Tracker before training with a clear intent: complete a planned session without spending time managing tools. She enters the workout area, starts a session, and logs each set quickly during rest intervals. The UI keeps interactions lightweight and predictable, so she does not need to context-switch away from training.

As she finishes, she ends the session and immediately sees a persisted workout entry in history, linked to her goal-tracking context. The key value moment is confidence that execution data has been captured with start/end session timestamps and integrated into the same system where her goals live. Her new reality is less admin overhead and more focus on training quality.

### Journey 2: Primary User (Athlete) - Edge Case and Recovery

During a busy gym session, Ana starts logging but gets interrupted, leaves the screen, or enters incomplete data for one set. On return, she expects the flow to be forgiving and easy to recover without losing momentum.

The product supports clear recovery behavior in the frontend flow: session state remains understandable, invalid fields are obvious, and she can still complete and close the session. Even when the flow is imperfect, the system keeps the outcome usable and prevents tracking from becoming a chore. The resolution is that minor disruption does not break adherence.

### Journey 3: Admin/Operations User - Configuration and Readiness

The product owner/developer (Ana in this phase) needs to ensure the integrated workout frontend is coherent, stable, and ready for backend replacement. She reviews page coverage, validates that mocked data paths support all critical states, and confirms data boundaries are isolated behind API-ready seams.

The critical moment is proving that frontend behavior is reliable enough for user validation now while remaining structurally ready for later API integration. The resolution is operational confidence: the module can be tested end-to-end in the UI today and migrated to real services in a follow-up backend PRD without large rewrites.

### Journey 4: Support/Troubleshooting User - Issue Investigation

When a user reports confusion (for example, uncertainty about whether a workout was saved), the support/troubleshooting actor reproduces the path through session start, set logging, and session end. They inspect visible history/graph effects and expected state transitions in the mocked frontend.

The product should make troubleshooting fast through consistent interaction patterns and transparent outcomes on each page. The resolution is quicker diagnosis and user guidance, reducing trust loss and ensuring users keep using the integrated goal-plus-workout flow.

### Journey Requirements Summary

The journeys reveal required capabilities:

- Fast session lifecycle flow (start, in-session logging, end, post-session confirmation).
- Clear workout-entry persistence feedback in the UI and visible linkage to goal context.
- Error-tolerant in-session UX with straightforward recovery for interruptions or incomplete input.
- Mobile-first interaction model suitable for gym usage (low-friction taps, readable state, quick navigation).
- Modular mocked data architecture with explicit replacement seams for future API integration.
- Consistent cross-page information model (history, plans, graphs, timer, exercises) to support troubleshooting and user trust.

## Web App Specific Requirements

### Project-Type Overview

The workout integration will be implemented as part of the existing single-page application architecture. Navigation across workout history, plans, graphs, timer, and exercise management remains within SPA routing to preserve fast in-app transitions and reduce interaction friction during active training contexts.

Browser support for this PRD scope targets current stable versions of Arc (Chromium), Chrome, Safari, and Firefox. Legacy browser compatibility and progressive enhancement for non-targeted browsers are out of scope for MVP.

### Technical Architecture Considerations

The frontend must preserve API-ready boundaries while using mocked data sources in this phase. Data access should be encapsulated so mocked providers can be replaced by backend API clients without changing page-level interaction logic, route structure, or component contracts.

Because SEO and real-time features are explicitly out of scope, architecture decisions should optimize for authenticated/productive in-app usage rather than crawlability or live synchronization complexity. State updates can rely on local mock-store patterns and deterministic refresh behaviors appropriate for frontend validation.

### Browser Compatibility Requirements

- Functional parity required on latest Arc (Chromium), Chrome, Safari, and Firefox.
- Core workout flows (session start/log/end, history visibility, plan interactions, graph navigation, timer usage) must behave consistently across target browsers.
- Visual and interaction regressions on target browsers block MVP completion.

### Performance and Navigation Requirements

- SPA route transitions between workout surfaces should feel immediate under normal local development conditions.
- Session logging interactions must avoid perceptible lag that disrupts in-gym usage.
- Page and interaction performance targets prioritize user flow continuity over advanced optimization work in this PRD.

### Accessibility Baseline Requirements

A minimum accessibility baseline is required for MVP:
- Semantic structure for primary interactive regions and forms.
- Keyboard-reachable controls for critical actions.
- Basic labeling for form fields and interactive elements.
- Sufficient contrast and readable text hierarchy in workout-critical views.

Formal WCAG conformance auditing is not required in this phase, but obvious accessibility blockers in core workflows must still be resolved before release.

### Implementation Considerations

Implementation should focus on reliable workout UX integration and maintainability over feature expansion. New code must align with existing frontend standards, keep mocked data explicit, and avoid introducing coupling that would complicate backend replacement in the follow-up PRD.

Out-of-scope items for this step include SEO optimization, real-time sync, legacy browser support, and advanced accessibility certification workflows.

## Project Scoping & Phased Development

### MVP Strategy & Philosophy

**MVP Approach:** Problem-solving MVP focused on validating reduced workout logging friction through a unified goal-plus-workout flow.  
**Resource Requirements:** Solo developer execution with staged delivery, strict phase boundaries, and explicit deferment of non-essential polish and backend-dependent features.

### MVP Feature Set (Phase 1)

**Core User Journeys Supported:**
- Primary athlete success path (start session, log sets, end session, see history linked to goals).
- Primary athlete edge-case recovery path (interruption/incomplete input with successful completion).
- Admin/ops validation path for integrated frontend quality and API-readiness.
- Support/troubleshooting path for reproducible session and history troubleshooting.

**Must-Have Capabilities:**
- Workout module pages integrated into current SPA navigation (history, plans, plan detail, graphs, graph detail, timer, exercise management).
- Mocked data implementation across all workout pages with API-ready seams (adapters/repositories/service boundaries).
- Session lifecycle UX with clear start/end timestamps and visible post-session entry effects in UI.
- Mobile-first usability baseline and minimum accessibility baseline for critical actions.
- Cross-browser support for latest Arc (Chromium), Chrome, Safari, and Firefox.
- No blocking lint/type errors introduced by this scope.

### Post-MVP Features

**Phase 2 (Post-MVP):**
- Backend integration and persistence replacement for all mocked workout data paths.
- Data synchronization and improved reliability semantics for multi-session usage.
- Enhanced analytics and richer trend/progression views.
- Deeper goal-workout linkage logic beyond basic session association.

**Phase 3 (Expansion):**
- Advanced progression intelligence and coaching-oriented feedback loops.
- Extended use cases, configurable workflows, and broader long-term adherence tooling.
- Platform-level capabilities that support scaling beyond initial solo-use workflow assumptions.

### Risk Mitigation Strategy

**Technical Risks:** High risk of coupling page components directly to mock data shapes. Mitigation: isolate data contracts through replaceable adapters/repositories, enforce stable view-model boundaries, and verify replacement readiness during implementation.  
**Market Risks:** Risk that users do not adopt the unified flow as expected. Mitigation: measure session completion and repeat usage in integrated surfaces, then iterate on interaction friction points before broadening scope.  
**Resource Risks:** Limited time and solo bandwidth may pressure scope expansion. Mitigation: enforce strict MVP boundary, defer non-essential enhancements to Phase 2/3, and prioritize workflows that directly validate the core hypothesis.

## Non-Functional Requirements

### Performance

- Core workout interactions (start session, log set, end session, navigate between workout pages) should complete with perceived responsiveness suitable for in-gym use, targeting sub-2-second completion for standard UI actions under normal client conditions.
- SPA route transitions across workout surfaces should remain smooth and non-blocking for target browsers.
- UI state updates from mocked providers should not introduce noticeable lag in critical logging flows.

### Security

- No sensitive credentials or secrets may be hardcoded in frontend workout modules or mock data layers.
- Data contracts and mock-provider boundaries must be structured to support secure backend replacement without exposing internal implementation details to page-level components.
- Frontend interactions must preserve existing app authentication/authorization assumptions and not bypass established protected-route behavior.

### Accessibility

- Critical workout flows must satisfy minimum accessibility baseline: keyboard-reachable primary controls, semantic structure for major regions/forms, and basic labeling for actionable controls/inputs.
- Text and control presentation in workout-critical screens must maintain readable contrast and hierarchy in supported themes.
- Obvious accessibility blockers in primary journeys (session start/log/end, plan management, history viewing) are considered release blockers.

### Integration

- All workout data interactions in Phase 1 must be routed through replaceable mock-provider seams (adapter/repository/service boundaries) to allow backend API substitution without page-level rewrites.
- Functional contracts between UI and data providers must remain stable and documented enough for backend PRD handoff.
- The integrated workout module must coexist with current goal-tracker frontend architecture and routing model without breaking existing goal flows.

### Reliability

- Core user journeys must produce consistent visible outcomes in UI state (for example, completed session appears in history context according to defined mock behavior).
- Error and recovery states in critical workflows must be explicit and recoverable without forcing full-app restart.
- Cross-browser functional parity is required for latest Arc (Chromium), Chrome, Safari, and Firefox on MVP-critical workflows.

### Scalability

- Phase 1 frontend architecture should avoid assumptions that prevent future scale-up (for example, tightly coupled page-state/data-shape dependencies), even though high-scale traffic optimization is out of current scope.
- The module should support incremental migration from local mocked data behavior to API-backed behavior without re-implementing core user journeys.
